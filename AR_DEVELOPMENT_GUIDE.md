# Manga AR 开发文档

## 1. 文档目的

本文档用于指导 `Manga-ar` 项目的逐步开发，目标是从当前可运行的 AR 工程骨架出发，最终实现一个可商用的 AR 摆放产品。

核心目标：

- 从后端获取可用模型列表
- 在 AR 场景中自由摆放一个或多个模型
- 支持移动、旋转、缩放，以及高度调节
- 支持拍照和录像
- 支持场景数据保存与恢复
- 为后续接入账号、作品管理、分享、审核等能力预留结构

本文档默认基于当前技术栈：

- `Expo 54`
- `React Native 0.81`
- `@reactvision/react-viro`
- Android 真机优先开发

---

## 2. 产品目标

### 2.1 最终产品形态

用户进入应用后，可以：

1. 从后端获取模型列表
2. 选择一个模型进入 AR 场景
3. 扫描真实环境平面
4. 点击平面放置模型
5. 对模型进行编辑：
   - 移动
   - 旋转
   - 缩放
   - 调整高度
6. 在同一场景摆放多个模型
7. 对整个 AR 场景拍照或录像
8. 保存场景，下次继续编辑

### 2.2 MVP 目标

第一阶段先做出可演示版本，满足以下最低要求：

1. 后端返回模型列表
2. 用户可选择一个模型
3. 用户可点击平面放置模型
4. 用户可调整位置、旋转、缩放、高度
5. 用户可拍照
6. 用户可录像

## 4. 推荐开发原则

### 4.1 模型格式建议

后端优先提供 `GLB` 模型。

原因：

- 单文件，最适合远程分发
- 资源管理简单
- 比 `OBJ + MTL + 贴图` 更适合移动端
- 接口返回更清晰

建议暂不把 `OBJ` 作为主交付格式，除非历史资源已经固定。

### 4.2 功能推进原则

开发顺序应遵循：

1. 先打通单模型完整链路
2. 再扩展多模型
3. 先支持基础编辑，再做复杂手势
4. 先做本地保存，再做后端同步

### 4.3 视觉与交互原则

AR 场景里必须尽量减少复杂 UI 覆盖，避免遮挡镜头画面。

建议：

- 主控制按钮放在屏幕底部
- 当前选中模型的编辑控件浮层显示
- 复杂操作拆成模式，而不是同时开放全部手势

## 8. 后端接口设计建议

以下为建议接口，不要求一次性全部实现。

### 8.1 获取模型列表

`GET /api/models`

返回示例：

```json
{
  "items": [
    {
      "id": "model_001",
      "name": "角色A",
      "thumbnailUrl": "https://example.com/thumbs/a.png",
      "modelUrl": "https://example.com/models/a.glb",
      "format": "GLB",
      "width": 1.2,
      "height": 1.8,
      "depth": 0.6,
      "defaultScale": 1,
      "tags": ["anime", "character"]
    }
  ]
}
```

### 8.2 获取模型详情

`GET /api/models/:id`

用途：

- 拉取更完整的模型配置
- 获取默认缩放、材质、展示参数

### 8.3 上传拍照结果

`POST /api/media/photo`

用途：

- 上传用户拍摄截图
- 返回可分享地址

### 8.4 上传录像结果

`POST /api/media/video`

用途：

- 上传用户录制的视频

### 8.5 保存场景

`POST /api/scenes`

请求示例：

```json
{
  "name": "我的第一个场景",
  "models": [
    {
      "id": "instance_1",
      "modelId": "model_001",
      "position": [0.12, 0.05, -1.2],
      "rotation": [0, 45, 0],
      "scale": [0.8, 0.8, 0.8],
      "heightOffset": 0.1
    }
  ]
}
```

### 8.6 获取历史场景

`GET /api/scenes/:id`

用途：

- 重新恢复之前摆放好的 AR 场景

## 10. 模型下载与缓存方案

### 10.1 推荐策略

采用“远程拉取元数据，本地缓存模型文件”的方案。

流程：

1. 获取模型列表
2. 用户选择模型
3. 检查本地缓存是否已存在
4. 若不存在，则下载到应用缓存目录
5. 把本地路径传给 `Viro3DObject`

### 10.2 为什么不建议每次直接在线加载

- 网络抖动会导致模型加载失败
- 首次展示时间不可控
- 场景里多个模型时更容易卡顿

### 10.3 缓存策略建议

- 以 `modelId + version` 为缓存 key
- 模型更新时服务端提升版本号
- 支持手动清缓存

## 11. AR 场景设计

### 11.1 推荐场景结构

AR 场景建议拆成以下层次：

1. 平面检测层
2. 模型实例渲染层
3. 当前选中模型编辑层
4. UI 控制层

### 11.2 场景职责

`ARPlacementScreen`

- 管理页面按钮
- 负责模型列表入口、拍摄入口、保存入口

`ARSceneView`

- 承载 `ViroARSceneNavigator`
- 提供截图、录像能力
- 向内部 Scene 传递模型实例和交互状态

`AR Scene`

- 检测平面
- 放置模型
- 更新模型位姿

## 12. AR 摆放交互设计

这是项目最核心的模块。

### 12.1 放置流程

1. 用户选择模型
2. 模型进入“待放置”状态
3. 场景扫描平面
4. 用户点击平面
5. 在点击位置创建模型实例
6. 新实例自动成为当前选中模型

### 12.2 推荐初版摆放方式

初版不要一开始就做非常复杂的手势编辑，推荐采用“模式化编辑”：

- 默认模式：点击放置
- 移动模式：拖拽平面内移动
- 旋转模式：按钮或滑杆旋转
- 缩放模式：按钮或滑杆缩放
- 高度模式：单独调节模型离地高度

这种方式比一次性上双指旋转缩放更稳定，开发成本更低，也更适合 AR 产品首版。

### 12.3 高度调节设计

你特别提出“能够调整模型的高度”，这部分建议单独设计，不能只靠普通平面拖拽。

建议采用以下数据结构：

```ts
position = [x, baseY + heightOffset, z]
```

解释：

- `baseY`：模型放到平面上的基础高度
- `heightOffset`：用户后续手动加上的抬升高度

这样做的好处：

- 模型仍然有“落在平面上”的基础逻辑
- 用户可以把模型轻微抬高，模拟悬浮、站台、飞行等效果
- 保存场景时只需要额外存一个 `heightOffset`

### 12.4 高度调节交互建议

建议优先使用底部滑杆，而不是直接用 3D 手势拖 Y 轴。

推荐交互：

1. 用户选中某个模型
2. 点击“高度”按钮
3. 底部出现高度滑杆
4. 滑杆范围例如 `-1.0m ~ +2.0m`
5. 拖动滑杆实时更新模型 Y 值

建议参数：

- 默认值：`0`
- 步进：`0.01m`
- 范围：`-0.5m ~ 2.0m`

后续可升级为两种模式：

- 精准模式：底部滑杆
- 快捷模式：上下微调按钮

### 12.5 模型移动

建议优先支持平面内拖拽：

- `dragType="FixedToWorld"` 或者结合命中检测更新位置
- 仅更新 `x/z`
- 不直接改 `heightOffset`

### 12.6 模型旋转

建议第一版使用：

- 左旋按钮
- 右旋按钮
- 角度滑杆

初版不强依赖双指旋转，避免误操作。

### 12.7 模型缩放

建议第一版使用：

- 放大按钮
- 缩小按钮
- 比例滑杆

缩放建议限制范围：

- `0.2` 到 `3.0`

## 13. 多模型管理设计

### 13.1 多实例原则

同一个模型资源可以在场景中创建多个实例。

区分方式：

- `modelId`：模型资源 ID
- `instanceId`：场景中的实例 ID

### 13.2 选中机制

同一时刻只允许一个模型实例处于选中状态。

选中后显示：

- 边框或高亮提示
- 编辑工具条
- 当前参数信息

### 13.3 删除机制

建议提供：

- 删除当前模型按钮
- 清空整个场景按钮

## 14. 拍照与录像设计

### 14.1 可用能力

`ViroARSceneNavigator` 已支持：

- `takeScreenshot`
- `startVideoRecording`
- `stopVideoRecording`

### 14.2 拍照流程

1. 用户点击“拍照”
2. 调用截图方法
3. 返回文件路径
4. 预览截图
5. 用户选择保存、上传、重拍

### 14.3 录像流程

1. 用户点击“开始录像”
2. UI 进入录制状态
3. 录制中隐藏复杂编辑控件
4. 用户点击“停止录像”
5. 生成视频文件
6. 进入预览页面

### 14.4 录制中限制

录像时建议限制以下操作：

- 不允许切换模型库
- 不允许删除场景
- 不建议同时复杂编辑模型

## 15. 场景保存与恢复

### 15.1 本地保存内容

建议至少保存：

- 场景 ID
- 模型实例列表
- 每个实例的模型 ID
- 本地模型缓存路径
- 位置信息
- 旋转信息
- 缩放信息
- 高度偏移

### 15.2 恢复策略

恢复场景时：

1. 读取场景数据
2. 检查模型缓存是否还在
3. 若不存在则重新下载
4. 重新渲染所有实例

### 15.3 注意事项

AR 世界坐标与用户真实空间高度相关，跨时间恢复时可能会存在偏差。

因此第一版建议：

- 保存“模型布局参数”
- 但不承诺完全恢复到真实世界原坐标

更实用的策略是：

- 进入场景后重新扫描平面
- 然后把历史布局重新相对摆放出来

## 17. 关键技术风险

### 17.1 模型过大导致卡顿

风险：

- 首次加载慢
- 内存占用高
- 帧率下降

策略：

- 服务端限制模型大小
- 约束面数与贴图尺寸
- 优先使用压缩后的 `GLB`

### 17.2 远程模型资源不完整

风险：

- `OBJ/GLTF` 附带资源路径丢失

策略：

- 第一版只支持 `GLB`

### 17.3 高度调节导致穿模或悬浮异常

风险：

- 模型底部原点不统一
- 视觉上像漂浮错误

策略：

- 服务端记录模型包围盒或底部原点
- 前端为每个模型配置默认落地偏移

### 17.4 多模型场景性能下降

策略：

- 限制场景同时摆放数量
- 对大模型做分级
- 提供“低性能模式”

## 26. 双端协作架构（AR + AI 工作流）

> 本章节记录 AR + 电脑端精调 + AI 漫画化 的整体技术方向与实施路径。

### 26.1 整体工作流

```
物理直觉（手机AR摆pose）
    → 空间数字化（实时同步3D场景至电脑端）
    → 精细创作（电脑端 Three.js 查看器 + 编辑面板）
    → 渲染输出（截图 / 序列帧 / 视频）
    → AI风格化（漫画 / 漫剧）
```

### 26.2 系统架构

```
手机端（Manga-ar · React Native + ViroReact）
┌─────────────────────────────────────────────┐
│  AR 场景                                    │
│  - 平面检测 & 模型摆放                       │
│  - 粗调：拖拽 / 旋转 / 缩放 / 摇杆           │
│  - 锁定状态显示（电脑端精调时模型半透明）      │
│  - syncService → WebSocket push 场景快照     │
└─────────────────┬───────────────────────────┘
                  │ WebSocket（JSON 场景快照）
                  ▼
         中继服务器 / Yjs Provider
                  │
                  ▼
电脑端（manga-ar-studio · React + Three.js）
┌─────────────────────────────────────────────┐
│  SceneViewer3D    — Three.js 实时 3D 查看    │
│  ModelInspector   — 右侧精调面板             │
│  RenderOutput     — 截图 / 序列帧输出         │
│  aiService        — 调用 SD + ControlNet API │
└─────────────────────────────────────────────┘
```

### 26.3 同步技术选型

| 方案 | 延迟 | 复杂度 | 适合阶段 |
|------|------|--------|----------|
| WebSocket + 中央服务器 | ~50ms | 低 | 起步首选 |
| WebSocket + Yjs（CRDT）| ~50ms | 低-中 | 推荐，天然解决冲突 |
| WebRTC Mesh | ~20ms | 中 | ≤4 设备 |
| WebRTC + SFU | ~20ms | 高 | 多设备扩展 |

**推荐起步方案：WebSocket + Yjs**，Yjs 天然解决冲突，后续可无缝升级到 WebRTC。

### 26.4 冲突解决：CRDT + 编辑锁

**核心问题**：手机和电脑同时修改同一模型时，谁赢？

**方案**：
- 使用 **Yjs** 作为 CRDT 层，自动合并无冲突的并发编辑
- 在模型实例上增加 `lockedBy?: 'phone' | 'desktop'` 字段
- 电脑端进入精细调整时对该模型加锁
- 手机端看到锁定状态 → 该模型显示为半透明 + 不可操作
- 防止电脑精调到一半被手机误触重置

**数据结构扩展**（`SceneModelInstance`）：

```ts
export type SyncLockOwner = 'phone' | 'desktop';

export type SceneModelInstance = {
  // ... 原有字段不变 ...
  lockedBy?: SyncLockOwner;   // 新增：编辑锁
  syncVersion?: number;       // 新增：乐观锁版本号
};
```

### 26.5 多设备坐标系统一

| 方案 | 推荐阶段 | 说明 |
|------|----------|------|
| **二维码空间锚点** | 起步首选 | 房间贴一个二维码作为原点，所有手机扫同一个 → 坐标系统一；离线可用，无后端依赖 |
| ARCore Cloud Anchors | 中期 | 无需物理标记，依赖 Google/Apple 服务 |
| SLAM 特征点融合 | 后期 | 最优雅，专业 AR 协作系统做法 |

### 26.6 手机↔电脑分工策略

**策略 A（推荐起步）：手机负责空间感知，电脑负责创作细节**
- 手机 AR：看到真实环境 + 模型大致位置（粗调层）
- 电脑端：看到精细调整后的完整场景
- 职责清晰，复杂度最低

**策略 B（后期升级）：完整状态同步**
- 手机 AR 实时显示电脑精调后的最终效果叠加在真实环境
- 更沉浸，但对手机渲染性能要求高

### 26.7 渲染输出策略

```
方案1: 电脑端实时渲染 → 截图/录屏
  优点：即时，所见即所得
  缺点：质量受显卡限制

方案2: 服务端离线渲染（Blender / Arnold）
  优点：高质量，支持全局光照、景深
  缺点：需要等待，流程更复杂

方案3: 混合——实时预览 + 按需高质量渲染（推荐）
```

**漫画输出的渲染预处理**（送给 AI 前先做风格化，效果更好）：
- Toon Shading（卡通着色）→ AI 漫画化成本大幅降低
- 线稿提取（Sobel 边缘检测）→ 给 AI 更清晰的结构信息
- 深度图 + 法线图 → 作为 AI 的 ControlNet 条件

### 26.8 导演模式（后期扩展）

> 3D 场景本身就是分镜脚本。

传统漫画：文字脚本 → 手绘分镜 → 上墨上色

本工作流可以变成：**在房间里「演」出场景 → 自动生成分镜 → AI 渲染成漫画**

- 在 3D 场景里定义摄像机路径和关键帧
- 系统自动按景别（远景 / 中景 / 特写）切割画格
- AI 渲染每一格 → 完整漫画页面

### 26.9 分阶段实施计划（双端协作部分）

#### 第一步：手机端推送（当前任务）

- 验证：手机摆模型，电脑浏览器实时同步显示

#### 第三步：双向同步 + 冲突处理
- 引入 Yjs，电脑端改动反向推送至手机
- 实现编辑锁（`lockedBy`）+ 手机端锁定态 UI

#### 第四步：渲染输出 + AI 漫画化

## 27. 后续开发文档（下一阶段执行版）

### 27.1 当前阶段判断

当前代码状态可以定义为：

- **AR MVP 主链路已打通**
- **多模型与基础编辑已完成**
- **拍照 / 录像已进入可联调阶段**

1. 把 mock 流程切换为真实业务流
2. 把“可运行”能力补成“可交付”能力
3. 把单机 MVP 补成带保存、同步、稳定性的产品原型

### 27.2 下一阶段总目标

接下来建议按 4 个重点方向并行推进：

#### 方向 A：真实数据接入

目标：

- 用真实后端接口替换当前 mock 模型列表
- 验证真实模型资源的下载、缓存、加载是否稳定

交付结果：

- App 启动后从真实接口拉取模型列表
- 可选择真实模型并进入 AR
- 下载失败、空列表、接口失败时有明确反馈

#### 方向 B：场景保存与恢复

目标：

- 建立场景 JSON 结构
- 支持本地保存当前摆放结果
- 支持从历史场景恢复编辑

交付结果：

- 用户退出后可以重新进入上次场景
- 至少支持本地存一份最近场景
- 恢复后模型位置、旋转、缩放、高度基本一致

#### 方向 C：拍照 / 录像稳定化

目标：

- 在真机上完整验证拍照与录像能力
- 统一失败提示、保存提示、预览流程

交付结果：

- 拍照稳定保存到相册
- 录像能开始、停止、预览、保存
- 录制中 UI 行为可控，不影响核心体验

#### 方向 D：双端协作闭环

目标：

- 打通手机端、relay server、电脑端工作台
- 让电脑端可以实时看到手机摆放结果
- 为后续双向编辑和 AI 工作流打地基

交付结果：

- 手机端场景快照能稳定推送
- 电脑端能实时渲染场景
- 至少完成单向同步可视化联调

### 27.3 推荐开发顺序

建议按以下顺序推进，而不是同时把所有事情铺开：

#### 第 1 步：接真实模型接口

原因：

- 这是所有真实业务联调的入口
- 不先接接口，后面的缓存、保存、同步都只能基于 mock 测试

具体任务：

- 在 `src/api/models.ts` 中接入真实接口地址
- 明确接口返回字段与当前 `RemoteModel` 的差异
- 若后端字段不一致，做最薄的一层转换
- 验证缩略图、模型地址、格式字段是否稳定

验收标准：

- 可以加载真实模型列表
- 接口异常时页面可重试
- 空数据时模型库页有明确空状态

#### 第 2 步：完成真实模型下载链路联调

原因：

- 当前缓存模块已可用，但还需要验证真实模型文件是否都能成功下载与加载

具体任务：

- 验证真实 `GLB` 模型是否都能被成功缓存
- 验证重复进入时是否命中本地缓存
- 补充下载失败时的提示文案
- 检查大模型的首次加载时间

验收标准：

- 同一模型不会每次重复下载
- 下载失败时不会卡死在加载状态
- 至少 3 个真实模型可稳定进入 AR

#### 第 3 步：补齐场景保存与恢复

原因：

- 当前场景实例结构已经具备，适合尽快完成本地持久化闭环

建议保存的数据：

```ts
{
```

具体任务：

- 在 AR 页面增加“保存当前场景”入口
- 增加“恢复最近场景”入口
- 恢复前检查模型缓存是否存在，不存在则重新下载

验收标准：

- 能保存当前场景
- 杀掉 App 后再次进入仍可恢复最近一次布局
- 恢复失败时有明确提示，不影响重新开始

### 27.4 拍照 / 录像稳定化计划

这部分建议单独作为一次真机回归任务处理。

#### 重点验证项

- 不同机型上截图是否成功
- 录像开始 / 停止是否返回正确结果
- 录制文件是否能预览与保存
- 没有权限时是否有清晰提示
- 录像中是否需要限制添加模型、删除模型等操作

#### 建议补齐的交互

- 拍照成功后的 toast / alert 文案统一
- 录像中显示明显的录制状态
- 录像结束后统一进入预览态
- 保存后清空临时预览状态

#### 验收标准

- 连续拍照 3 次不崩溃
- 连续录像 2 次流程正常
- 相册中可看到生成的照片和视频

### 27.5 双端协作下一步计划

当前手机端推送能力已经具备，下一阶段应重点做电脑端接收和可视化。

#### 电脑端最小版本目标

建议电脑端第一版只做 3 个能力：

1. 建立 WebSocket 连接
2. 接收 `scene_snapshot`
3. 在 Three.js / React Three Fiber 中渲染模型占位结果

#### 为什么先做“查看器”而不是“编辑器”

- 先确认同步链路稳定比先做精调面板更重要
- 查看器做通后，再决定双向同步和编辑锁细节
- 可快速验证坐标系、缩放、朝向等基础问题

#### 电脑端阶段任务

- 在 `manga-ar-studio/` 中初始化页面入口
- 建立 `syncClient`
- 接收手机端快照并转成前端状态
- 渲染模型实例或临时几何体占位
- 显示当前选中实例信息

验收标准：

- 手机端移动或新增模型时，电脑端能实时看到变化
- 电脑端断线重连后能重新收到最新快照

说明：

- `sceneStorage.ts` 是下一阶段最值得优先新增的模块
- `captureService.ts` 不是必须立即抽离，但如果拍照录像逻辑继续增多，可以再收拢
- `scene.ts` 用来区分“运行时实例结构”和“持久化结构”

### 27.7 下一阶段任务拆解（可直接执行）

#### Sprint A：真实接口与资源联调

- 接入真实模型列表接口
- 对齐接口字段
- 联调真实缩略图与模型地址
- 验证缓存命中与下载失败提示

#### Sprint B：场景保存与恢复

- 定义 `SceneDocument`
- 新增本地存储 service

#### Sprint C：拍照 / 录像稳定化

- 整理拍摄流程状态
- 补充异常提示

#### Sprint D：双端协作首轮闭环

- 跑通 relay server

- 渲染实时场景
- 验证局域网同步链路

### 27.8 风险与注意事项

#### 1. 真实模型资源质量不稳定

风险：

- 模型过大
- 原点不统一
- `surfaceOffset` 缺失
- 某些资源虽然可下载但无法正常显示

建议：

- 先筛选一批可控样例作为联调白名单
- 接口返回中尽量带上默认缩放和落地偏移

#### 2. 场景恢复并不等于真实世界锚点恢复

风险：

- 用户重新进入后环境识别结果变化
- 同样的数值位置未必对应现实中的同一位置

建议：

- 第一版只承诺“恢复布局参数”
- 不承诺完全恢复到现实世界绝对位置

#### 3. 双端同步先解决“看得到”，再解决“改得对”

建议：

- 第一轮只做单向同步
- 第二轮再做编辑锁和反向更新
- 不要过早引入复杂冲突处理

### 27.9 推荐验收里程碑

#### 里程碑 M1：真实业务流跑通

- 真实模型列表可用
- 真实模型可下载并进入 AR
- 至少 3 个模型联调通过

#### 里程碑 M2：场景闭环完成

- 当前场景可保存
- 重新进入可恢复最近布局

#### 里程碑 M3：拍摄能力稳定

- 真机拍照 / 录像流程稳定
- 相册保存成功率可接受

#### 里程碑 M4：双端可视化打通

- 手机摆放结果可同步到电脑端
- 电脑端可实时显示模型状态变化

### 27.10 本章节结论

从现在开始，项目的推进策略应从“继续补功能点”切换为“补闭环 + 做联调 + 做稳定性”。

优先级建议固定为：

1. **真实接口接入**
2. **场景保存与恢复**
3. **拍照 / 录像稳定化**
4. **双端协作联调**

只要这 4 个方向补齐，当前项目就会从“可演示的 AR MVP”进入“可持续迭代的产品原型”阶段。
