# Manga AR 开发文档

## 1. 文档目的

本文档用于指导 `Manga-ar` 项目的逐步开发，目标是从当前可运行的 AR 工程骨架出发，最终实现一个可商用的 AR 摆放产品。

核心目标：

- 从后端获取可用模型列表
- 在 AR 场景中自由摆放一个或多个模型
- 支持移动、旋转、缩放，以及高度调节
- 支持拍照和录像
- 支持场景数据保存与恢复
- 为后续接入账号、作品管理、分享、审核等能力预留结构

本文档默认基于当前技术栈：

- `Expo 54`
- `React Native 0.81`
- `@reactvision/react-viro`
- Android 真机优先开发

---

## 2. 产品目标

### 2.1 最终产品形态

用户进入应用后，可以：

1. 从后端获取模型列表
2. 选择一个模型进入 AR 场景
3. 扫描真实环境平面
4. 点击平面放置模型
5. 对模型进行编辑：
   - 移动
   - 旋转
   - 缩放
   - 调整高度
6. 在同一场景摆放多个模型
7. 对整个 AR 场景拍照或录像
8. 保存场景，下次继续编辑

### 2.2 MVP 目标

第一阶段先做出可演示版本，满足以下最低要求：

1. 后端返回模型列表
2. 用户可选择一个模型
3. 用户可点击平面放置模型
4. 用户可调整位置、旋转、缩放、高度
5. 用户可拍照
6. 用户可录像

## 4. 推荐开发原则

### 4.1 模型格式建议

后端优先提供 `GLB` 模型。

原因：

- 单文件，最适合远程分发
- 资源管理简单
- 比 `OBJ + MTL + 贴图` 更适合移动端
- 接口返回更清晰

建议暂不把 `OBJ` 作为主交付格式，除非历史资源已经固定。

### 4.2 功能推进原则

开发顺序应遵循：

1. 先打通单模型完整链路
2. 再扩展多模型
3. 先支持基础编辑，再做复杂手势
4. 先做本地保存，再做后端同步

### 4.3 视觉与交互原则

AR 场景里必须尽量减少复杂 UI 覆盖，避免遮挡镜头画面。

建议：

- 主控制按钮放在屏幕底部
- 当前选中模型的编辑控件浮层显示
- 复杂操作拆成模式，而不是同时开放全部手势

## 8. 后端接口设计建议

以下为建议接口，不要求一次性全部实现。

### 8.1 获取模型列表

`GET /api/models`

返回示例：

```json
{
  "items": [
    {
      "id": "model_001",
      "name": "角色A",
      "thumbnailUrl": "https://example.com/thumbs/a.png",
      "modelUrl": "https://example.com/models/a.glb",
      "format": "GLB",
      "width": 1.2,
      "height": 1.8,
      "depth": 0.6,
      "defaultScale": 1,
      "tags": ["anime", "character"]
    }
  ]
}
```

### 8.2 获取模型详情

`GET /api/models/:id`

用途：

- 拉取更完整的模型配置
- 获取默认缩放、材质、展示参数

### 8.3 上传拍照结果

`POST /api/media/photo`

用途：

- 上传用户拍摄截图
- 返回可分享地址

### 8.4 上传录像结果

`POST /api/media/video`

用途：

- 上传用户录制的视频

### 8.5 保存场景

`POST /api/scenes`

请求示例：

```json
{
  "name": "我的第一个场景",
  "models": [
    {
      "id": "instance_1",
      "modelId": "model_001",
      "position": [0.12, 0.05, -1.2],
      "rotation": [0, 45, 0],
      "scale": [0.8, 0.8, 0.8],
      "heightOffset": 0.1
    }
  ]
}
```

### 8.6 获取历史场景

`GET /api/scenes/:id`

用途：

- 重新恢复之前摆放好的 AR 场景

## 10. 模型下载与缓存方案

### 10.1 推荐策略

采用“远程拉取元数据，本地缓存模型文件”的方案。

流程：

1. 获取模型列表
2. 用户选择模型
3. 检查本地缓存是否已存在
4. 若不存在，则下载到应用缓存目录
5. 把本地路径传给 `Viro3DObject`

### 10.2 为什么不建议每次直接在线加载

- 网络抖动会导致模型加载失败
- 首次展示时间不可控
- 场景里多个模型时更容易卡顿

### 10.3 缓存策略建议

- 以 `modelId + version` 为缓存 key
- 模型更新时服务端提升版本号
- 支持手动清缓存

## 12. AR 摆放交互设计

这是项目最核心的模块。

### 12.1 放置流程

1. 用户选择模型
2. 模型进入“待放置”状态
3. 场景扫描平面
4. 用户点击平面
5. 在点击位置创建模型实例
6. 新实例自动成为当前选中模型

### 12.2 推荐初版摆放方式

初版不要一开始就做非常复杂的手势编辑，推荐采用“模式化编辑”：

- 默认模式：点击放置
- 移动模式：拖拽平面内移动
- 旋转模式：按钮或滑杆旋转
- 缩放模式：按钮或滑杆缩放
- 高度模式：单独调节模型离地高度

这种方式比一次性上双指旋转缩放更稳定，开发成本更低，也更适合 AR 产品首版。

### 12.3 高度调节设计

你特别提出“能够调整模型的高度”，这部分建议单独设计，不能只靠普通平面拖拽。

建议采用以下数据结构：

```ts
position = [x, baseY + heightOffset, z]
```

解释：

- `baseY`：模型放到平面上的基础高度
- `heightOffset`：用户后续手动加上的抬升高度

这样做的好处：

- 模型仍然有“落在平面上”的基础逻辑
- 用户可以把模型轻微抬高，模拟悬浮、站台、飞行等效果
- 保存场景时只需要额外存一个 `heightOffset`

### 12.4 高度调节交互建议

建议优先使用底部滑杆，而不是直接用 3D 手势拖 Y 轴。

推荐交互：

1. 用户选中某个模型
2. 点击“高度”按钮
3. 底部出现高度滑杆
4. 滑杆范围例如 `-1.0m ~ +2.0m`
5. 拖动滑杆实时更新模型 Y 值

建议参数：

- 默认值：`0`
- 步进：`0.01m`
- 范围：`-0.5m ~ 2.0m`

后续可升级为两种模式：

- 精准模式：底部滑杆
- 快捷模式：上下微调按钮

### 12.5 模型移动

建议优先支持平面内拖拽：

- `dragType="FixedToWorld"` 或者结合命中检测更新位置
- 仅更新 `x/z`
- 不直接改 `heightOffset`

### 12.6 模型旋转

建议第一版使用：

- 左旋按钮
- 右旋按钮
- 角度滑杆

初版不强依赖双指旋转，避免误操作。

### 12.7 模型缩放

建议第一版使用：

- 放大按钮
- 缩小按钮
- 比例滑杆

缩放建议限制范围：

- `0.2` 到 `3.0`

## 13. 多模型管理设计

### 13.1 多实例原则

同一个模型资源可以在场景中创建多个实例。

区分方式：

- `modelId`：模型资源 ID
- `instanceId`：场景中的实例 ID

### 13.2 选中机制

同一时刻只允许一个模型实例处于选中状态。

选中后显示：

- 边框或高亮提示
- 编辑工具条
- 当前参数信息

### 13.3 删除机制

建议提供：

- 删除当前模型按钮
- 清空整个场景按钮

## 14. 拍照与录像设计

### 14.1 可用能力

`ViroARSceneNavigator` 已支持：

- `takeScreenshot`
- `startVideoRecording`
- `stopVideoRecording`

### 14.2 拍照流程

1. 用户点击“拍照”
2. 调用截图方法
3. 返回文件路径
4. 预览截图
5. 用户选择保存、上传、重拍

### 14.3 录像流程

1. 用户点击“开始录像”
2. UI 进入录制状态
3. 录制中隐藏复杂编辑控件
4. 用户点击“停止录像”
5. 生成视频文件
6. 进入预览页面

### 14.4 录制中限制

录像时建议限制以下操作：

- 不允许切换模型库
- 不允许删除场景
- 不建议同时复杂编辑模型

## 15. 场景保存与恢复

### 15.1 本地保存内容

建议至少保存：

- 场景 ID
- 模型实例列表
- 每个实例的模型 ID
- 本地模型缓存路径
- 位置信息
- 旋转信息
- 缩放信息
- 高度偏移

### 15.2 恢复策略

恢复场景时：

1. 读取场景数据
2. 检查模型缓存是否还在
3. 若不存在则重新下载
4. 重新渲染所有实例

### 15.3 注意事项

AR 世界坐标与用户真实空间高度相关，跨时间恢复时可能会存在偏差。

因此第一版建议：

- 保存“模型布局参数”
- 但不承诺完全恢复到真实世界原坐标

更实用的策略是：

- 进入场景后重新扫描平面
- 然后把历史布局重新相对摆放出来

## 17. 关键技术风险

### 17.1 模型过大导致卡顿

风险：

- 首次加载慢
- 内存占用高
- 帧率下降

策略：

- 服务端限制模型大小
- 约束面数与贴图尺寸
- 优先使用压缩后的 `GLB`

### 17.2 远程模型资源不完整

风险：

- `OBJ/GLTF` 附带资源路径丢失

策略：

- 第一版只支持 `GLB`

### 17.3 高度调节导致穿模或悬浮异常

风险：

- 模型底部原点不统一
- 视觉上像漂浮错误

策略：

- 服务端记录模型包围盒或底部原点
- 前端为每个模型配置默认落地偏移

### 17.4 多模型场景性能下降

策略：

- 限制场景同时摆放数量
- 对大模型做分级
- 提供“低性能模式”

## 18. 分线开发计划总览

从当前阶段开始，项目需要明确拆成三条相对独立的开发线：

1. **移动端 App**：负责 AR 空间感知、模型摆放、粗调、拍照录像、本地保存与移动端独立可用。
2. **PC 端 Studio**：负责桌面端查看、精调、渲染输出、AI 漫画化工作流和创作型功能。
3. **中继 / 同步服务**：负责移动端与 PC 端之间的可选连接、消息转发、会话隔离和后续协作控制。

核心原则：

- 移动端必须可以独立启动、独立摆放、独立保存、独立拍摄。
- PC 端不能成为移动端启动和使用的前置依赖。
- PC 端可以在任意时刻连接当前移动端会话，并获取最新场景状态。
- 中继服务只负责连接与同步，不承载移动端核心业务状态。
- 双端开发可以并行，但必须共享统一的场景数据协议。

## 22. 共享数据协议

移动端和 PC 端必须共享同一套同步协议。当前协议在：

```text
src/types/sync.ts
```

后续如果 PC 端工程独立出来，应复制或抽成共享包，避免两端各自定义一套字段。

### 22.1 核心实体

运行时模型实例核心字段：

```ts
{
  instanceId: string,
  asset: CachedModelAsset,
  x: number,
  y: number,
  z: number,
  rotationY: number,
  scaleValue: number
}
```

同步模型实例额外字段：

```ts
{
  lockedBy?: 'phone' | 'desktop',
  syncVersion: number
}
```

### 22.2 核心消息

当前已经定义：

- `scene_snapshot`：移动端推送完整场景快照
- `instance_update`：单个实例增量更新
- `instance_delete`：删除实例
- `lock_acquire`：申请编辑锁
- `lock_release`：释放编辑锁
- `ping` / `pong`：心跳

### 22.3 协议演进原则

- 新增字段必须向后兼容
- 不要随意重命名已有字段
- PC 端不应该依赖移动端内部 UI 状态
- 移动端不应该依赖 PC 端是否在线
- `sessionId` 是双端连接的最小隔离单位
- `instanceId` 是模型实例同步的主键

## 25. 编码与架构约定

### 25.1 移动端约定

- 移动端是 AR 主状态拥有者
- 所有模型实例状态统一由 `ARPlacementScreen` 或后续状态层管理
- `ModelPlacementScene` 尽量只负责 AR 渲染和交互事件，不直接写业务请求
- 模型资源请求和文件缓存必须分层封装
- 本地场景保存结构与运行时结构分离
- 移动端不得因为 PC 端未连接而阻塞核心功能
- 第一版默认只支持 `GLB`

### 25.2 PC 端约定

- PC 端是创作增强端，不是移动端启动依赖
- PC 端应通过 `scene_snapshot` 重建场景，而不是读取移动端内部状态
- PC 端修改实例必须发送标准 `instance_update`
- 真实模型加载失败时应回退到占位模型
- AI 能力必须作为独立服务封装，不应耦合 Three.js 查看器

### 25.3 中继服务约定

- 中继服务只做连接、房间和消息转发
- 不把中继作为移动端唯一状态存储
- 不在中继里写复杂业务逻辑
- 所有消息必须带 `sessionId` 或由连接上下文补齐
- 后续云端部署时再增加鉴权、持久化和多用户权限

### 25.4 同步协议约定

- `instanceId` 是模型实例的唯一主键
- `modelId` 是模型资源的唯一主键
- `syncVersion` 用于丢弃过期更新
- `lockedBy` 用于双端编辑锁
- 新增消息类型前必须同时更新移动端、PC 端和文档

## 26. 双端协作架构（AR + AI 工作流）

### 26.1 整体工作流

```text
移动端独立 AR 摆放
    → 可选连接 relay
    → PC 端随时接入当前 session
    → 桌面端精细创作
    → 渲染输出
    → AI 漫画化 / 漫剧化
```

### 26.2 系统架构

```text

┌─────────────────────────────────────────────┐
│  AR 场景                                    │
│  - 平面检测 & 模型摆放                       │
│  - 粗调：拖拽 / 旋转 / 缩放 / 摇杆           │
│  - 本地保存 / 恢复 / 拍照 / 录像             │
│  - syncService 可选连接 WebSocket relay      │
└─────────────────┬───────────────────────────┘
                  │ WebSocket（可选连接）
                  ▼
         relay-server（轻量中继）
                  │
                  ▼
电脑端（manga-ar-studio · Three.js / React）
┌─────────────────────────────────────────────┐
│  SceneViewer3D    — 桌面 3D 查看             │
│  ModelInspector   — 精调面板                 │
│  RenderOutput     — 截图 / 序列帧输出         │
│  aiService        — 漫画化 API 调用           │
└─────────────────────────────────────────────┘
```

### 26.3 手机端与 PC 端职责边界

移动端负责：

- AR 空间感知
- 模型粗摆放
- 本地保存和恢复
- 拍照录像
- 推送当前场景快照
- 接收 PC 端精调后的实例更新

PC 端负责：

- 大屏查看
- 精细编辑
- 渲染输出
- AI 漫画化工作流
- 后续分镜、镜头、导演模式

中继负责：

- 按 session 转发消息
- 缓存最后快照
- 支持后接入端快速恢复状态
- 后续处理连接保活、编辑锁广播和基础鉴权

### 26.4 同步技术路线

起步方案：

- WebSocket + relay-server
- JSON 消息协议
- 移动端推送 `scene_snapshot`
- PC 端发送 `instance_update`

后续升级：

- 引入 Yjs / CRDT 处理复杂并发
- 引入编辑锁减少冲突
- 引入 WebRTC 或云端部署支持远程协作

不要过早引入复杂协作系统。第一阶段目标是：**手机独立可用，PC 随时接入，双端状态能同步**。

### 26.5 渲染输出策略

```text
方案 1：PC 端实时渲染 → 截图 / 录屏
  优点：实现快、所见即所得
  缺点：质量受本机性能限制

方案 2：服务端离线渲染
  优点：质量高
  缺点：需要排队和等待

方案 3：实时预览 + 按需高质量渲染
  推荐作为中期目标
```

AI 漫画化前可在 PC 端预处理：

- Toon Shading
- 线稿提取
- 深度图
- 法线图
- 分层渲染

### 26.6 后期扩展：导演模式

后续 PC 端可以把 3D 场景发展为分镜工具：

- 定义摄像机路径和关键帧
- 按景别自动切分画格
- 输出多张分镜图
- 调用 AI 漫画化生成漫画页面

## 27. 当前结论

当前项目已经具备一个可演示的移动端 AR MVP，并且已经出现 PC 端 Studio 和 relay 的雏形。后续开发不应再把所有能力混在一条阶段线上推进，而应明确拆成：

1. **移动端独立产品线**：保证 AR 摆放、保存、拍摄、真实模型使用稳定。
2. **PC 端 Studio 产品线**：保证桌面查看、精调、渲染输出、AI 工作流逐步成型。
3. **中继 / 同步服务产品线**：保证双端可以随时连接、断开、重连和交换标准消息。

当前最重要的方向是：

- 移动端继续稳定化，不依赖 PC 端。
- PC 端从单文件 demo 演进为正式 Studio 工程。
- 中继服务补齐稳定连接与消息校验。
- 双端共享协议，避免各自实现一套不兼容的数据结构。

只要这三条线按上述边界推进，项目就能从“可演示 AR MVP”进入“移动端 + PC 创作端”的可持续产品原型阶段。
